<objective>
Empirically calibrate the Taiwan intraday cumulative-volume-weight curve from real 5-minute Yahoo data of ~50 large-cap stocks (0050 成分股近似) over 60 trading days, and replace the hand-written three-segment linear `getTaiwanVolumeWeight` in `utils/volume.ts` with a lookup-table linear interpolation over the calibrated array. Purpose: The current `getTaiwanVolumeWeight` is a guess (25% / 55% / 13% by hand). A wrong weight at minute t makes `projectedVolume = currentVolume / weight(t)` wrong, which feeds the 攻擊量 (attack-volume ≥1.3×) judgment in `runEntryFilter` step 5/SOP④. Real data gives a defensible curve and a defensible early-window guard. US curve is untouched.

2. Per symbol, fetch TWO Yahoo chart endpoints (base `https://query2.finance.yahoo.com/v8/finance/chart/`):
   - 5m series: `?interval=5m&range=60d&includePrePost=false`
   - 1d series (for daily totals): `?interval=1d&range=3mo`
   Send a browser-like `User-Agent` header. Run sequentially with a ~300ms delay between every request (rate-limit courtesy). Tolerate per-symbol failures: wrap each fetch in try/catch, log `skip <symbol>: <reason>`, continue. After the loop, REQUIRE ≥40 symbols fully succeeded (both endpoints parsed); if fewer, print a clear BLOCKER message to stderr and `process.exit(1)` — do not write a low-sample curve.

3. Cache raw JSON to `scripts/.cache-volume-calib/<symbol>-<interval>.json` (create dir if missing). On re-run, read from cache instead of re-fetching when the cache file exists. This dir is gitignored (see .gitignore step).

4. Build the time grid: 5-minute bar-start labels from 09:05 to 13:25 plus a final point 13:30. Represent grid by MINUTES-from-09:00: 5,10,15,…,265 (that's 09:05…13:25) and 270 (13:30). The cumulative weight at 270 is defined as 1.0 after adding the closing-auction leftover.

5. Per stock-day reconstruction (iterate each trading day present in the 5m series, keyed by Taipei date):
   - Group that day's 5m bars onto the grid by bar-start time. Use the bar's timestamp converted to Asia/Taipei; do NOT assume the 09:00 bar holds the opening auction (it may be volume 0).
   - sum_5m = sum of that day's 5m bar volumes.
   - daily_total = that date's volume from the 1d series. If the date is absent in the 1d series → drop the stock-day.
   - leftover = daily_total − sum_5m. Attribute leftover to the 13:30 closing-auction point.
   - cumulative share at each grid point t = (running sum of 5m volume up to and including t) / daily_total; the 13:30 point = (sum_5m + leftover) / daily_total = 1.0.
   - EXCLUSIONS (log a count per reason): bar count ≠ 54±1; sum_5m > daily_total (leftover negative); sum_5m < 50% of daily_total; half-trading days (implied by abnormal bar count); 期指結算日 (3rd Wednesday of each month — compute, don't hardcode). Drop excluded stock-days.

6. Aggregate across all surviving stock-days (expect ~1500–2000): at each grid point take the MEDIAN cumulative share; also compute p25 and p75.

7. Projection-error analysis: for each surviving stock-day and each grid point t, projected = cumVol(t)/medianWeight(t); error = projected/daily_total − 1. Per grid point compute error p10 and p90. Find the EARLIEST grid time T* (minutes from 09:00) where the p10..p90 band is within ±35%. T* is the empirical TW Insufficient cutoff.

8. Output:
   (a) Write `scripts/tw-volume-curve-report.md` (Traditional-Chinese headings consistent with repo bilingual style) containing: methodology summary, symbols attempted/succeeded, total surviving stock-days, exclusion counts by reason, the median curve table (minute | median | p25 | p75), the error-by-time table (minute | p10 | p90), and the chosen T* with its minute value.
   (b) Print to stdout the final ready-to-paste TypeScript const: `const TW_CUM_WEIGHT: number[] = [ … ];` — exactly 55 numbers for minutes 5,10,…,270 (index i ↔ minute (i+1)*5), last value 1.0, each rounded to 4 decimals. Also print `T* = <minutes>`.

Run the script with `node scripts/calibrate-tw-volume-curve.mjs` and capture the REAL output. The 55-value array and T* printed here are consumed verbatim by Task 2 — do not fabricate or round-trip through guesses. Re-runs use the cache, so a second run is cheap.
