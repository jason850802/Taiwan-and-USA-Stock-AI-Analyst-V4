<objective>
修正 API 後端 serverless function 在 Vercel production 部署時的 `ERR_MODULE_NOT_FOUND` 崩潰。

根因：`package.json` 為 `"type": "module"`，Vercel 以 Node 原生 ESM 直接執行編譯後的 `api/**/*.js`（不經 bundler）；原生 ESM 規則要求相對 import 必須帶明確副檔名。但原始碼在 `tsconfig.json` 的 `moduleResolution: "bundler"` 下允許無副檔名相對 import（此設定假設有 bundler 會改寫路徑——前端 Vite 有，但 api 沒有）。因此 4 個端點在模組載入階段即拋錯，handler 根本沒機會執行，全數 500 `FUNCTION_INVOCATION_FAILED`。

修法（TypeScript + Node ESM 標準慣用法）：在 6 個檔案的所有相對 import specifier 補上 `.js` 副檔名。即使原始檔是 `.ts`，import 字串仍寫 `.js`——`moduleResolution: "bundler"` 會正確把 `.js` 指回 `.ts` 原始碼。**只改 import 字串，不動任何邏輯，不動 `tsconfig.json`／`package.json`。**

Purpose: 讓後端代理層在真實 Vercel Lambda runtime 可正常載入並執行（本機 `vercel dev` 路徑解析較寬鬆，故此 bug 從 Phase 1 起潛伏至今未被抓到）。
Output: 6 個檔案共 16 處相對 import specifier 補上 `.js`。

備註（重要）：交辦描述標題寫「15 處」，但實際列舉清單與 grep 皆為 **16 處**（api/_lib/http.ts 的 type-only import 也算在內）。以 grep 結果為準——目標是「這 6 檔內每一個相對 import 都帶 .js」，不論最終數字。
</objective>
